# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

SBON is a Japanese-first SBOM viewer for non-engineers (procurement, QA, security, regulatory). It reads CycloneDX JSON / SPDX JSON, explains OSS components in Japanese, and surfaces review priority, vulnerabilities, license, and maintenance concerns. It ships as a **dependency-free static site** — the public deployment is a "トライアル版" (trial) at https://sbon.osamusic.org/.

## Commands

- **Run tests:** `npm test` (runs `node tests/run-tests.js`)
- **Run the app:** open `index.html` directly in a browser — there is no build step. For browser automation/screenshots, serve over HTTP first: `python3 -m http.server 8765` then load `http://localhost:8765/index.html`.

There is no bundler, linter, or transpiler. Keep it that way: **no runtime dependencies, no network calls for SBOM data.** SBOMs are processed entirely in the browser and never sent to a server — this privacy guarantee is a product promise, not just an implementation detail.

## Architecture

Vanilla JS with no module system. Each `js/*.js` file is an IIFE that attaches a single global (`window.SBON_PARSER`, `SBON_REVIEW_PRIORITY`, `SBON_REVIEW`, `SBON_DIFF`, `SBON_EXPORT`, `SBON_UI`). `app.js` is the only glue: it calls `SBON_UI.createViewer({ parser, exporter, differ, review, ... })` and `viewer.start()`.

**Script load order matters** and is hand-maintained in two places that must stay in sync:
- the `<script>` tags at the bottom of `index.html`
- the file list in `tests/run-tests.js` `loadCore()`

Knowledge-base data loads first, then samples, then the `js/` modules. A module referencing another's global before it is defined will break.

Data flow when an SBOM is loaded:
1. `js/parser.js` normalizes CycloneDX/SPDX into a common component shape, collecting identifiers (purl, CPEs) and matching each component to a knowledge-base `packageId`. Matching tries purl name, CPE product, and aliases (`data/knowledge/package-identifiers.js`); the winning strategy is recorded as `matchMethod` (e.g. `cpe-product`). This matching logic is the most subtle part of the codebase and is the focus of most tests.
2. Matched components are enriched with Japanese descriptions, categories, and review-priority via `js/review-priority.js` and the knowledge base.
3. `js/review.js` tracks per-component review state (未確認/確認中/承認/要対応) + notes, persisted to `localStorage`, exportable/importable as JSON.
4. `js/diff.js` compares two SBOMs (added/removed/changed, priority escalation, license changes, dependency diffs).
5. `js/export.js` produces print/PDF report rows and UTF-8 **BOM-prefixed** CSV (so Excel opens it correctly — do not drop the BOM).
6. `js/ui.js` (`createViewer`) owns all DOM wiring, filtering, sorting, pagination, tabs, and bulk review actions.

## Knowledge base — do not hand-edit

`data/knowledge/*.js` (packages, identifiers, categories, entries-ja, review-priority-rules) plus `data/knowledge-base.js` are the core IP. **They are generated by a separate private repo and exported here.** Treat them as build artifacts: do not hand-edit data in this repo; change the generator instead. `data/knowledge/knowledge-meta.js` carries the `version` shown in the site footer and the entity counts.

## Tests

`tests/run-tests.js` is a custom runner with **no test framework** — plain functions using `node:assert`, registered in the `run()` list at the bottom of the file. It loads the real `js/`/`data/` modules into a mock `window` via `vm`, and provides hand-rolled DOM + `localStorage` mocks for the UI tests.

- Add a test by writing a `testXxx()` function and adding it to the `run()` list.
- There is no per-test filter flag; to run one in isolation, temporarily trim the `run()` list or call the function from a one-off `node -e` script.
- When you touch parser matching or knowledge-base shape, expect `testMultiCpeMatching`, `testAliasMatching`, `testCpeProductExtraction`, and `testSpdxSampleEnrichment` to be the relevant guards.
