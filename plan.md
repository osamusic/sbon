# SBON Development Plan

## 現状

SBONは、非エンジニアのSBOM利用者向けに、CycloneDX JSON / SPDX JSONを読み込んで日本語で確認できる最小限の静的SBOM Viewerとして動作している。

現在できること:

- CycloneDX JSON / SPDX JSONの読み込み
- コンポーネント一覧の表示
- 依存関係の簡易表示
- コンポーネント詳細の日本語説明
- 脆弱性、保守確認、ライセンス未確認などに基づく確認優先度表示
- 検索、確認優先度絞り込み、レビュー観点絞り込み
- ブラウザ印刷によるPDF出力
- CSV出力

## 方針

当面は「SBOMを受け取った非エンジニアが、まず何を確認すべきか分かる」状態を優先する。脆弱性スキャンツールやCI/CD向け機能ではなく、レビュー、説明、調達・品質保証・セキュリティ管理の意思決定支援に寄せる。

## Phase 1: 静的MVPの完成度向上

目的: 依存なしでブラウザだけで使える状態を保ちながら、入力対応と表示品質を上げる。

### 0. 知識ベース設計: 完了

- 説明辞書を単純に増やさない
- `knowledge-base-design.md` に基づき、DB移行可能なデータモデルを先に固める
- 当面は静的JSONで運用し、将来SQLiteへ移行できる形にする
- パッケージ名だけでなく、PURL、CPE、エコシステム、ディストリビューション名、別名を識別子として使って照合する

### 1. SBOMパーサの強化: 未完

- CycloneDX 1.4 / 1.5 / 1.6 の主要フィールドに対応
- SPDX 2.2 / 2.3 の主要フィールドに対応
- package URL、CPE、supplier、publisher、copyrightの表示
- SPDX relationshipの表示精度向上
- SBOMファイルの形式判定エラーを分かりやすくする

### 2. 確認優先度判定の見直し: 初期完了

- 現在の簡易ヒューリスティックを明示的なルールセットに分離
- 確認理由を「脆弱性」「保守」「ライセンス」「用途不明」に分類
- 重大度を High / Medium / Low / Unknown で一貫表示
- EOLや脆弱性の確定判定ではなく、確認優先度として扱う
- 外部DB連携までは行わない

### 3. 日本語知識ベースの初期構築: 初期完了

- `knowledge-base-design.md` のモデルに沿って、説明、識別子、カテゴリ、確認優先度ルールを分離する
- OpenSSL、BusyBox、glibc、Linux kernelの4件だけを新形式へ移す
- 各コンポーネントに「何に使われるか」「なぜ重要か」「確認すべきこと」「根拠」を持たせる
- curl、nginx、Apache HTTP Server、OpenSSH、zlib、SQLite、Python、Node.jsなどは、スキーマと照合ロジックが固まってから追加する

### 4. レポート品質の改善: 着手中

- 印刷時に要約、確認優先度一覧、詳細が読みやすいPDFレイアウトにする
- CSVに確認理由、用途説明、PURL、supplierを含める
- 「経営・調達向け要約」と「技術確認向け詳細」を分ける

進捗:

- CSV出力は確認優先度、照合情報、脆弱性ID、説明文を含む形に改善済み
- 印刷用レポートの要約セクションは追加済み
- PDF専用の詳細レイアウトは未完

### 5. UIの使い勝手改善: 着手中

- コンポーネント一覧の列幅、ソート、件数表示を追加
- フィルタ条件のリセットボタンを追加
- 選択中コンポーネントが絞り込みで消えた場合の表示を自然にする
- 空状態、読み込み失敗、形式未対応の表示を改善

進捗:

- 一覧の件数表示は追加済み
- 名称、バージョン、確認優先度、用途、ライセンス、脆弱性数のソートは追加済み
- フィルタ解除ボタンは追加済み
- UI操作テストは追加済み
- 空状態、読み込み失敗、選択状態の改善は未完

## Phase 2: テストと保守性

目的: 実SBOMを安全に扱えるように、壊れにくい構成へ移す。

### 1. テストデータ整備

- CycloneDXサンプルを複数追加
- SPDXサンプルを複数追加
- 脆弱性あり、依存関係あり、ライセンス不明、巨大ファイルのケースを用意

### 2. 自動テスト

- SBOM正規化処理のユニットテスト
- 確認優先度判定ルールのユニットテスト
- CSV出力のテスト
- 最小限のブラウザ表示テスト

### 3. ファイル構成の整理

- `app.js` を parser / review-priority / render / export に分割
- 説明・識別子・カテゴリ・確認優先度ルールを `data/knowledge/` 配下のJSONへ移動
- サンプルSBOMを `samples/` に移動

## Phase 3: アプリ化

目的: 単体HTMLから、継続開発しやすいアプリケーションへ移行する。

### 1. 技術スタック選定

候補:

- Vite + React + TypeScript
- Vite + Vue + TypeScript
- 静的HTML + TypeScriptビルド

初期推奨は Vite + React + TypeScript。理由は、画面状態、詳細パネル、フィルタ、将来の比較ビューを整理しやすいため。

### 2. 型定義

- NormalizedComponent
- NormalizedSbom
- Vulnerability
- License
- DependencyNode
- RiskFinding

### 3. 状態管理

- 最初はReact標準stateで十分
- 大規模化するまでは外部状態管理ライブラリを入れない

## Phase 4: 実用機能

目的: 実際のレビュー業務で使える機能を増やす。

### 1. SBOM Diff Viewer

- 2つのSBOMを読み込み
- 追加、削除、更新されたコンポーネントを表示
- 確認優先度が上がった項目を強調
- リリースレビュー向け要約を生成

### 2. レビューコメント

- コンポーネントごとに確認メモを残す
- ステータスを 未確認 / 確認中 / 承認 / 要対応 で管理
- JSONとしてレビュー結果を保存、再読み込み

### 3. レポート出力

- PDF専用レイアウト
- Excel出力
- 調達向け概要レポート
- 技術レビュー向け詳細レポート

## Phase 5: 外部情報連携

目的: SBOM内情報だけでは不足する、脆弱性・保守・ライセンス情報を補完する。

### 1. 脆弱性情報

- OSV
- NVD
- GitHub Security Advisory

### 2. 保守情報

- release-monitoring.org
- GitHub Releases
- endoflife.date

### 3. ライセンス情報

- SPDX License List
- ClearlyDefined

注意: 外部連携はネットワーク、キャッシュ、誤検知、利用規約、監査ログの設計が必要になるため、静的MVPの後に扱う。

## Phase 6: AI支援

目的: 非エンジニアが確認事項の意味を理解しやすくする。

### 1. 説明支援

- 「このコンポーネントは何か」
- 「この脆弱性は事業上どう問題になるか」
- 「調達時にベンダーへ何を確認すべきか」

### 2. レビュー支援

- 重要コンポーネントの自動抽出
- 不明点の質問案生成
- 監査・規制対応向けの説明文生成

注意: AI出力は判断の補助に限定し、根拠となるSBOM項目・脆弱性情報・ルールを必ず併記する。

## 優先順位

直近で着手する順番:

1. `app.js` の責務分割: 完了
2. 知識ベースJSONの初期スキーマ作成: 完了
3. 確認優先度への用語・内部モデル整理: 完了
4. 一覧UIの基本操作改善: 着手中
5. PDF/CSV出力の改善: 着手中
6. サンプルSBOMの追加
7. パーサ強化
8. 空状態・エラー表示・選択状態の改善
9. SBOM Diff Viewerの設計

## 完了条件

短期の完了条件:

- 代表的なCycloneDX/SPDXサンプルを読み込める
- 非エンジニアが確認理由を日本語で理解できる
- 調達・品質保証・セキュリティ管理向けの要約を出せる
- CSVまたはPDFでレビュー結果を共有できる
- パーサと確認優先度判定が自動テストで保護されている

中期の完了条件:

- SBOM差分をレビューできる
- コンポーネントごとのレビュー状態を保存できる
- 外部脆弱性情報と突き合わせられる
- レポートを業務利用できる品質で出力できる
